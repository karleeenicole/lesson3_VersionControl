VCS Version Control System

A Version Control System is a system used for backing up files. Its used for the purpose of recovering and modifying previous versions of a file. It is very useful when working on a large project with several people contributing (often simultaneously). A VCS solves the file sharing issue by utilizing a main common repository allowing team members to checkout, edit, commit, and merge files. It comes in handy when one wants to save a current version of a file that they may want to revert back to. The system keeps track of what exact changes were made by who and when. They are very smart to use in case of a mistake the system holds backups.

Git Let's git it

GIT is a leading Version Control System that is known as distributed version control. This is a system that allows users to check out the fully repository and history, not just the latest version of the file. This is a better system because if the server goes down, all is not lost and clients can back up the entire repository. This is different from a centralized system because the version database each person is working with allows for complete range of motion for working on the project in large capacity. GIT is fast, simple, and fully distributed for large projects to be easily maintained between and across branches.

GitHub Using it effectively

GitHub is the largest host for Git repositories and the main point of collaboration between millions of projects and developers.

  1. The first step is to create an account on GitHub.
  2. Next, configure a public key to use SSH remotes by clicking on "Settings" -> "Add an SSH key"
  3. Paste the contents of your public-key and click "Add key."
  4. Add your email address and two-factor authentication if wanted.
  5. Next you'll need to obtain a git repository. This can be done by taking a local directory that isn't under a VCS and put it under Git, or clone an existing Git repository from somewhere else.
  6. Go to the project directory "$ cd /Users/user/my_project" and type "$ git init"
  7. This creates a subdirectory named ".git" (a skeleton).
  8. Next we'll go over the git commands for navigating the system.

Git Commands Usin(g) it

"$ git init nameofproject"
this is the initialization of the project, it sets up the git control structures. It creates a folder directory or local repository that contains the project files and history.
$ git add filename
this command permanately recognizes the files as ready to be committed into the directory. It places it into a temporary holding area primed to be committed which must be done first. This is known as staging. A file must be staged before it can be committed.
$ git commit -m "Commit message"
Git commit is the command that actually saves and records a version of the file into the directory, but it is not yet added to the remote repository.
$ git commit -a
This commits any files added with git add and any files modified since then.
$ git push origin master
This sends changes to the master branch of the remote repository.
$ git status
This displays all the files modified and those still needed to be added or committed.
git checkout -b branchname
Creates a new branch name and switches to it.

Repositories Where are they

It can get confusing understanding the locations of files with adding, committing, and pushing, but Git does a clear job of keeping it straight. Remote and local repositories are confusing concepts. In the local directory is your working directory, staging area, and local repo. Things are placed into the local repo when they are committed. They are placed into the staging area when they are added. Git push places a file into the remote repo. Git pull is used to pull it back from the remote repo to local workspace. Where git checkout places it back into the working directory from the local repo. The remote repo is only involved in git push and git pull. The concept are easier to visualize when mapped out.
